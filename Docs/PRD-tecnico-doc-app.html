1Visão técnica

Módulo Documentos (canal App) é o "Despachante digital" do RODE COM LUCRO: hub privado onde o caminhoneiro autônomo cadastra e mantém em dia documentos legais (CNH, CRLV/RENAVAM, RNTRC/ANTT, tacógrafo/aferição, seguro obrigatório) e documentos sem vencimento (CPF/CNPJ, placa, comprovante de rastreador) com data de vencimento, status calculado e dias para vencer. O motorista cadastra/edita datas (o sistema recalcula status e diasParaVencer), faz upload de foto/PDF para storage PRIVADO (acesso só por URL assinada expirável) e recebe push em marcos (30/15/7/1 dia e no dia). O módulo expõe um STATUS AGREGADO (ex.: "documentação em dia", "1 doc vencendo", "1 doc vencido") SEM o arquivo nem datas detalhadas, para alimentar o score de embarcador no portal B2B (INTERESSADOS[].score e selo de carga 'validada'). O protótipo é 100% mockado: src/data/documentos.ts (type StatusDoc string fixa) e src/data/pendencias.ts (StatusPendencia 'ok'|'proximo'|'vencido' + diasParaVencer, já usado na faixa de aviso do Início via PROXIMO_VENCIMENTO/CONTAGEM_PENDENCIAS). ATENÇÃO: pendencias.ts mistura categoria 'documento' e 'manutencao'; este PRD cobre SOMENTE categoria 'documento' — a faixa de aviso do Início continua mesclando ambas no front (manutenção fica fora de escopo). Este PRD troca os mocks de documentos por backend real (Supabase/Postgres + RLS + Edge Functions + Storage privado + cron). OCR fica para fase posterior — aqui a data de vencimento é digitada. Princípio central: o cálculo de status é determinístico e centralizado no banco (função rode_doc_status) para que App, job de push e score-agregado leiam a MESMA verdade. NOTA DE RECONCILIAÇÃO: o mock atual marca RNTRC com 34 dias como 'proximo', divergindo de um cutoff fixo de 30 dias; o PRD adota cutoff parametrizável por tipo_documento.dias_alerta (default 30) para preservar a UX do mock sem hardcode.

2Arquitetura

  • Fonte única de verdade do status: status e dias_para_vencer NÃO são persistidos como texto livre; são derivados de data_vencimento por uma função SQL determinística rode_doc_status(data_vencimento, dias_alerta) STABLE usada pela view vw_documentos_status e pela RPC de leitura. Elimina divergência entre App, push e score (corrige o mock atual onde status era string fixa em documentos.ts).
  • Cutoff de alerta parametrizável: tipo_documento.dias_alerta (default 30) define a janela 'proximo' por tipo, reconciliando o mock (RNTRC a 34 dias aparece como 'proximo'). rode_doc_status recebe dias_alerta do catálogo, evitando hardcode.
  • Backend Supabase/Postgres com Row Level Security: cada documento tem motorista_id = auth.uid(); políticas RLS por comando (SELECT/INSERT/UPDATE/DELETE) garantem que o motorista só lê/escreve os próprios documentos. Edge Functions/cron usam service_role com bypass de RLS apenas em jobs server-side.
  • Storage privado por padrão: bucket 'documentos' NÃO público; arquivos sob caminho documentos/{motorista_id}/{documento_id}/{uuid}.{ext}. Download só via createSignedUrl com TTL curto (60-300s) gerado por Edge Function que valida ownership. Nenhuma URL pública é gravada no banco.
  • Separação App vs B2B por contrato: o portal de embarcadores nunca acessa a tabela documento; consome só a RPC get_status_agregado_documentos(motorista_id) (SECURITY DEFINER) que retorna rótulo agregado + contagens, sem datas e sem caminho de arquivo. Fronteira de dados explícita para LGPD (minimização).
  • Notificações push via Web Push (PWA, VAPID) + tabela push_subscription; um cron diário (pg_cron ou Supabase Scheduled Function) detecta marcos 30/15/7/1/0 e enfileira notificações idempotentes (UNIQUE documento+marco+vencimento em notificacao_log).
  • Camada de API fina como Edge Functions (Deno/TS) para operações sensíveis (assinar URL upload/download, finalizar upload, agregação, push); CRUD simples de datas vai direto via supabase-js + RLS, reduzindo superfície de código.
  • Mobile-first PWA React+TS (Vite): tela Documentos reusa o modelo diasParaVencer/StatusPendencia de pendencias.ts; estado em React Query (cache + revalidação) com optimistic update ao salvar data e rollback em erro. A faixa de aviso do Início mescla documentos (API) + manutenção (mock, fora de escopo) mantendo PROXIMO_VENCIMENTO.

3Stack sugerida

Frontend: React 18 + TypeScript + Vite (PWA mobile-first, já existente no repo). Estado de servidor com @tanstack/react-query; formulários com react-hook-form + zod. Ícones lucide-react (FileCheck já em uso em papeis.ts). Service Worker (Workbox) para Web Push e cache offline.Backend: Supabase (Postgres 15) com Row Level Security; Edge Functions em Deno/TypeScript para assinatura de URL, finalização de upload, agregação e envio de push.Storage: Supabase Storage bucket privado 'documentos' com signed URLs (createSignedUploadUrl / createSignedUrl).Banco: Postgres + extensões pgcrypto (gen_random_uuid) e pg_cron (agendamento) — ou Supabase Scheduled Functions como alternativa se pg_cron indisponível.Push: Web Push API com chaves VAPID; biblioteca web-push (Deno) na Edge Function de envio; chaves VAPID em secrets do projeto, nunca no client.Auth: Supabase Auth (telefone/OTP, coerente com público caminhoneiro); JWT com auth.uid() amarrado a motorista_id.Observabilidade: Supabase Logs + tabelas documento_evento (auditoria) e notificacao_log; advisors de segurança (get_advisors) rodados antes de cada release.Testes: Vitest (unit front + funções SQL via pgTAP/SQL asserts), supabase-js em testes de integração contra branch efêmero, Playwright para e2e PWA (incluindo mock de Service Worker push).

4Modelo de dados

tipo_documento (catálogo)

  • codigo text PK ('cnh'|'crlv'|'rntrc'|'tacografo'|'seguro'|'cpf_cnpj'|'placa'|'rastreador')
  • nome text NOT NULL
  • tem_vencimento boolean NOT NULL
  • obrigatorio boolean NOT NULL DEFAULT true (entra no cálculo de cobertura do agregado)
  • dias_alerta int NOT NULL DEFAULT 30 (janela 'proximo')
  • orgao text NULL ('DETRAN'|'ANTT'|'INMETRO'|'SUSEP')
  • ordem int NOT NULL. Seed alinhado a documentos.ts + pendencias.ts: CNH(tem_venc=true), CRLV/RENAVAM(true), RNTRC/ANTT(true), Tacógrafo(true), Seguro obrigatório(true), CPF/CNPJ(false), Placa(false), Comprovante de rastreador(false).

documento

  • id uuid PK default gen_random_uuid()
  • motorista_id uuid NOT NULL refs auth.users(id) ON DELETE CASCADE
  • tipo_codigo text NOT NULL refs tipo_documento(codigo)
  • numero text NULL
  • data_vencimento date NULL
  • arquivo_path text NULL (caminho no bucket, nunca URL)
  • arquivo_mime text NULL CHECK (arquivo_mime IN ('image/jpeg','image/png','application/pdf'))
  • arquivo_tamanho_bytes int NULL CHECK (arquivo_tamanho_bytes <= 10485760)
  • criado_em timestamptz NOT NULL default now()
  • atualizado_em timestamptz NOT NULL default now()
  • CHECK (data_vencimento IS NULL OR tipo_codigo IN (SELECT codigo FROM tipo_documento WHERE tem_vencimento))
  • UNIQUE(motorista_id, tipo_codigo). status e dias_para_vencer NÃO são colunas físicas — derivados na view. Índice idx_documento_motorista ON documento(motorista_id).

vw_documentos_status (view)

  • Projeta documento JOIN tipo_documento + status text = rode_doc_status(data_vencimento, dias_alerta) ∈ {'em_dia','proximo','vencido','sem_data'}
  • dias_para_vencer int (data_vencimento - current_date, NULL se sem data)
  • proximo_marco int (próximo de 30/15/7/1/0 ainda futuro, NULL se vencido/sem_data)
  • tem_arquivo boolean (arquivo_path IS NOT NULL)
  • nome, ordem, tem_vencimento, obrigatorio. NÃO projeta arquivo_path. Mapeia para StatusPendencia do front ('em_dia'->'ok','proximo'->'proximo','vencido'->'vencido','sem_data'->'ok' sem alerta).

documento_evento (auditoria)

  • id bigint PK GENERATED ALWAYS AS IDENTITY
  • documento_id uuid refs documento(id) ON DELETE CASCADE
  • motorista_id uuid NOT NULL
  • tipo text NOT NULL CHECK (tipo IN ('data_alterada','arquivo_enviado','arquivo_removido','visualizado','excluido_lgpd'))
  • payload jsonb (valor antigo/novo, ex.: {de:'2027-03-01', para:'2026-07-01'})
  • criado_em timestamptz NOT NULL default now(). Índice ON (motorista_id, criado_em DESC).

push_subscription

  • id uuid PK default gen_random_uuid()
  • motorista_id uuid NOT NULL refs auth.users(id) ON DELETE CASCADE
  • endpoint text NOT NULL UNIQUE
  • p256dh text NOT NULL
  • auth text NOT NULL
  • user_agent text NULL
  • criado_em timestamptz NOT NULL default now()
  • ultimo_ok_em timestamptz NULL (atualizado em envio bem-sucedido
  • subscriptions com 410 Gone são removidas).

notificacao_log (idempotência de marcos)

  • id bigint PK GENERATED ALWAYS AS IDENTITY
  • documento_id uuid NOT NULL refs documento(id) ON DELETE CASCADE
  • motorista_id uuid NOT NULL
  • marco int NOT NULL CHECK (marco IN (30,15,7,1,0))
  • data_vencimento_ref date NOT NULL
  • enviado_em timestamptz NOT NULL default now()
  • status_envio text NOT NULL CHECK (status_envio IN ('enviado','falhou'))
  • tentativas int NOT NULL DEFAULT 1
  • UNIQUE(documento_id, marco, data_vencimento_ref) — garante 1 push por documento+marco+vencimento.

5Contratos de API

GET/rest/v1/vw_documentos_status?motorista_id=eq.{uid}&order=ordem

Via supabase-js + RLS (herda RLS da tabela base documento). Lista documentos do motorista logado com status derivado, dias_para_vencer, proximo_marco e tem_arquivo. Resposta: [{id, tipo_codigo, nome, numero, data_vencimento, status:'em_dia'|'proximo'|'vencido'|'sem_data', dias_para_vencer, proximo_marco, tem_arquivo:boolean, tem_vencimento, obrigatorio, ordem}]. NUNCA expõe arquivo_path. 401 se sem JWT; retorna [] (não erro) se motorista sem documentos.

POST/rest/v1/documento (upsert, on_conflict=motorista_id,tipo_codigo)

Cria ou edita numero/data_vencimento. Payload: {tipo_codigo, numero?, data_vencimento?}. RLS WITH CHECK força motorista_id=auth.uid(); cliente NÃO envia motorista_id (default via trigger/policy). Trigger BEFORE UPDATE grava documento_evento 'data_alterada' (de/para) e atualiza atualizado_em. Rejeita 400 se data_vencimento informada para tipo com tem_vencimento=false (CHECK). Resposta 200/201 com a linha; o front refaz a query da view para obter status novo. Idempotente por (motorista_id, tipo_codigo).

POST/functions/v1/documentos-upload-url

Edge Function. Payload: {documento_id|tipo_codigo, mime, tamanho_bytes}. Valida JWT, ownership (documento pertence ao auth.uid()), mime∈{image/jpeg,image/png,application/pdf} e tamanho_bytes∈(0,10485760], gera caminho documentos/{uid}/{doc_id}/{gen_random_uuid()}.{ext} (ext derivada do mime, nunca do nome do cliente) e retorna {signed_upload_url, token, arquivo_path} via createSignedUploadUrl (TTL 120s). 400 mime/tamanho inválido; 403 ownership; 404 documento inexistente. Cliente faz PUT direto no Storage.

POST/functions/v1/documentos-finalizar-upload

Edge Function. Payload: {documento_id, arquivo_path, mime, tamanho_bytes}. Valida que arquivo_path tem prefixo documentos/{uid}/{documento_id}/ (anti-path-traversal), confirma objeto no Storage (HEAD), grava arquivo_path/mime/tamanho no documento, apaga arquivo anterior (se houver), registra evento 'arquivo_enviado'. Resposta: {ok:true, tem_arquivo:true}. 409 se objeto não existe no Storage; 403 se prefixo não bate com uid.

POST/functions/v1/documentos-download-url

Edge Function. Payload: {documento_id}. Valida ownership via JWT, exige arquivo_path NOT NULL (404 se sem arquivo), registra evento 'visualizado', retorna {signed_url, expira_em} com createSignedUrl TTL 180s. Único caminho para abrir o arquivo. 403 se documento de outro motorista.

DELETE/functions/v1/documentos-arquivo

Edge Function. Payload: {documento_id}. Valida ownership, remove objeto do Storage, limpa arquivo_path/mime/tamanho no banco, registra 'arquivo_removido'. Resposta: {ok:true, tem_arquivo:false}. Idempotente (204 mesmo se já sem arquivo).

POST/rest/v1/rpc/get_status_agregado_documentos

RPC SECURITY DEFINER consumida pelo portal B2B/score. Param: {p_motorista_id uuid}. Retorna SOMENTE agregado: {rotulo:'em_dia'|'pendente'|'vencido', total int, vencidos int, proximos int, sem_data_obrigatorios int, atualizado_em timestamptz}. NUNCA retorna datas, números, arquivo_path ou tipo individual. Base do selo 'documentação OK' das cargas validadas. GRANT EXECUTE apenas a role de serviço B2B (não a anon/authenticated genérico do App).

POST/functions/v1/push-subscribe

Edge Function. Payload: {endpoint, keys:{p256dh, auth}, user_agent}. Upsert (on_conflict=endpoint) em push_subscription para o motorista logado (motorista_id=auth.uid()). Resposta 201. 400 se keys ausentes.

POST/functions/v1/lgpd-excluir-documentos

Edge Function. Payload: {confirmacao:true}. Apaga todos objetos do Storage sob documentos/{uid}/, deleta linhas de documento do motorista (cascade em evento/notificacao), grava documento_evento 'excluido_lgpd' (sem payload sensível). Resposta: {ok:true, removidos:int}. Atende direito de exclusão LGPD art. 18.

6Frontend / fluxo

  • Tela DocumentosHub (rota /documentos, alvo do papel 'despachante' em papeis.ts telaAlvo='documentos'): cards de documento ordenados por urgência (vencidos primeiro, reusando a lógica de PENDENCIAS_ORDENADAS — sort por dias_para_vencer asc, sem_data por último). Card: nome, faixa de status colorida (verde 'em dia' / âmbar 'vence em N dias' / vermelho 'vencido há N dias' / cinza 'sem data'), data de vencimento formatada (mar/2027), miniatura/ícone de arquivo (FileCheck). Faixa de aviso no topo replica PROXIMO_VENCIMENTO (item mais urgente, mesclando manutenção do mock).
  • Componente DocumentoCard: estados visuais derivados de status + dias_para_vencer (mapeamento StatusPendencia 'ok'|'proximo'|'vencido'); botões 'Editar data' e 'Ver/Enviar foto'. Tipos sem vencimento (CPF/CNPJ, placa, rastreador) escondem botão de data e faixa de prazo.
  • Sheet/Modal EditarVencimento: react-hook-form + zod (data_vencimento date >= hoje-3650 e <= hoje+3650, numero opcional max 40 chars). Ao salvar faz optimistic update no React Query e revalida vw_documentos_status para refletir o status recalculado; rollback + toast de erro em falha. Bloqueia campo data para tipo tem_vencimento=false.
  • Fluxo UploadDocumento: input câmera/galeria/PDF -> valida mime/tamanho no cliente (<=10MB, jpeg/png/pdf) -> documentos-upload-url -> PUT no Storage com barra de progresso -> documentos-finalizar-upload -> invalida query. Erro de validação client-side antes de qualquer chamada de rede.
  • VisualizarArquivo: ao tocar 'Ver foto' chama documentos-download-url e abre a signed URL em visor in-app (img/pdf viewer); nunca persiste a URL no estado React além do TTL nem em localStorage.
  • Integração de banner e badge: faixa de aviso do Início e badge do papel Despachante leem o agregado (count de não-'ok', equivalente a CONTAGEM_PENDENCIAS) agora vindo de vw_documentos_status e não do mock; manutenção permanece via pendencias.ts até fase futura.
  • Estado offline: React Query persistido (persistQueryClient) cacheia a última vw_documentos_status para leitura offline read-only; mutations de data ficam em fila e sincronizam ao reconectar.

7Lógica de negócio

  • Função SQL determinística rode_doc_status(p_venc date, p_dias_alerta int default 30) -> text STABLE: se p_venc IS NULL -> 'sem_data'; d := p_venc - current_date; d < 0 -> 'vencido'; d <= p_dias_alerta -> 'proximo'; senão 'em_dia'. dias_para_vencer := (p_venc - current_date). Mapeamento front: 'em_dia'->'ok','proximo'->'proximo','vencido'->'vencido','sem_data'->'ok'.
  • Cutoff por tipo: a view passa tipo_documento.dias_alerta a rode_doc_status, então RNTRC (dias_alerta>=34) aparece 'proximo' a 34 dias, reconciliando pendencias.ts; demais usam default 30.
  • Marcos de push: MARCOS=[30,15,7,1,0]. Para cada documento com data_vencimento e tem_vencimento=true, marco_devido quando dias_para_vencer ∈ MARCOS exatamente. Idempotência via UNIQUE(documento_id, marco, data_vencimento_ref): o cron só envia se ainda não há registro para o trio (evita repetição em reexecução ou edição de data — nova data gera novo data_vencimento_ref e novos marcos).
  • Recalculo on-write: como status é derivado na view, editar data_vencimento não exige escrita de status; a próxima leitura já reflete. Trigger só grava auditoria 'data_alterada' (de/para em payload) e atualiza atualizado_em.
  • Agregação para score (get_status_agregado_documentos): rotulo='vencido' se EXISTS doc vencido; senão 'pendente' se EXISTS doc 'proximo' OU algum tipo obrigatório (obrigatorio=true, tem_vencimento=true) sem documento/sem data; senão 'em_dia'. 'em_dia' exige cobertura completa dos obrigatórios, não só ausência de vencidos. atualizado_em = max(documento.atualizado_em).
  • Caminho de arquivo: documentos/{motorista_id}/{documento_id}/{gen_random_uuid()}.{ext}; ext derivada do mime no servidor; ao reenviar, novo objeto é criado e o anterior apagado em finalizar-upload (1 arquivo corrente por documento, sem lixo no bucket).
  • TTL de signed URL: upload 120s, download 180s; respostas com Cache-Control no-store; cada visualização gera nova URL e grava evento 'visualizado' para trilha LGPD.

8Integrações

  • Supabase Storage (bucket privado 'documentos') via createSignedUploadUrl/createSignedUrl — único meio de upload e download.
  • Web Push API (PWA) com chaves VAPID; biblioteca web-push na Edge Function de envio; Service Worker registra a subscription e exibe a notificação.
  • pg_cron (ou Supabase Scheduled Functions) para o job diário de detecção de marcos e enfileiramento de notificações.
  • Supabase Auth (telefone/OTP) fornecendo auth.uid() para RLS e ownership.
  • Portal B2B / módulo de score de embarcador (src/data/empresa.ts INTERESSADOS[].score, src/data/embarcadores.ts) como consumidor da RPC agregada para o selo de carga 'validada'.
  • Faixa de aviso do Início e badge do papel Despachante (src/data/papeis.ts, src/data/pendencias.ts) consumindo o agregado da API para documentos, mantendo manutenção no mock.
  • Integração futura (fora de escopo desta fase): OCR para extrair automaticamente a data de vencimento da foto enviada.

9Segurança & LGPD

  • RLS obrigatória em documento, documento_evento, push_subscription, notificacao_log: políticas por comando USING (motorista_id = auth.uid()) para SELECT/UPDATE/DELETE e WITH CHECK (motorista_id = auth.uid()) para INSERT/UPDATE; nenhum acesso anônimo (REVOKE de anon). vw_documentos_status herda RLS da tabela base.
  • Bucket 'documentos' privado; políticas de Storage permitem ao usuário operar apenas objetos sob prefixo documentos/{auth.uid()}/. Download exclusivamente por signed URL de TTL curto emitida por Edge Function que revalida ownership pelo JWT. createSignedUploadUrl também restrito a prefixo do uid.
  • RPC get_status_agregado_documentos é a ÚNICA superfície exposta ao portal B2B/score e retorna apenas rótulo agregado e contagens — sem datas, números de documento ou caminho de arquivo (minimização de dados, LGPD art. 6 II/III). GRANT EXECUTE restrito à role de serviço B2B; revogado de authenticated genérico.
  • Validação de upload no servidor (não confiar no cliente): mime allowlist {jpeg,png,pdf}, tamanho ∈ (0, 10MB], rejeição de extensões executáveis; nome do objeto gerado pelo servidor com gen_random_uuid e ext derivada do mime para evitar path traversal; finalizar-upload valida prefixo documentos/{uid}/{documento_id}/.
  • LGPD: dados de documento (CNH, CPF/CNPJ, placa) são dados pessoais; base legal = execução de serviço solicitado pelo titular (art. 7 V). Direito de exclusão (art. 18): Edge Function lgpd-excluir-documentos apaga objetos do Storage + linhas e grava log 'excluido_lgpd'. Retenção: arquivo apagado ao remover documento ou encerrar conta (ON DELETE CASCADE + limpeza de Storage); documento_evento guarda quem/quando acessou.
  • Auditoria de acesso: toda emissão de download-url grava documento_evento 'visualizado' com motorista_id e timestamp; trilha para responder solicitações do titular e detectar abuso (acessos anômalos).
  • Edge Functions validam JWT (Supabase Auth getUser) e ownership antes de qualquer assinatura de URL; service_role key fica em secrets, nunca exposta ao cliente; rate-limit básico por uid nas funções de upload/download para mitigar enumeração.

10Não-funcionais

  • Performance: tela Documentos carrega <1.5s em 3G médio (payload da view <5KB, sem arquivos); listagem usa idx_documento_motorista. Signed URL emitida em <300ms (p95).
  • Confiabilidade do push: cron diário idempotente (UNIQUE em notificacao_log) garante no máximo 1 notificação por documento+marco+vencimento mesmo com reexecução; falhas marcadas 'falhou' e reprocessáveis sem duplicar; subscriptions retornando 410 Gone são removidas automaticamente.
  • Disponibilidade offline: PWA cacheia a última vw_documentos_status (status) para leitura offline; edições ficam em fila (React Query mutation queue) e sincronizam ao reconectar; conflito resolvido por last-write-wins na chave (motorista_id, tipo_codigo).
  • Escalabilidade do cron: job processa em lote apenas documentos com dias_para_vencer ∈ MARCOS no dia (filtro indexado), evitando varredura total; suporta a base ~1.000 motoristas do critério MVP sem degradar (<10s por execução).
  • Segurança de storage: nenhuma URL pública persistida; TTL curto (<=180s) limita a janela de vazamento de link; Cache-Control no-store nas signed URLs.
  • Consistência: status sempre derivado em tempo de leitura via rode_doc_status — impossível App, push e score divergirem do banco.
  • Acessibilidade: cores de status acompanhadas de rótulo textual ('vencido há N dias'), contraste AA, alvos de toque >=44px (público caminhoneiro, uso na estrada).

  1. 1. Modelagem e migrations: criar tipo_documento (seed CNH/CRLV/RNTRC/tacografo/seguro com tem_vencimento=true + CPF-CNPJ/placa/rastreador com tem_vencimento=false), documento (com CHECK de data só para tipos com vencimento), documento_evento, push_subscription, notificacao_log; função rode_doc_status; view vw_documentos_status; índices.

  3. 3. CRUD de datas no App: tela DocumentosHub + DocumentoCard + EditarVencimento (react-hook-form/zod, React Query optimistic), lendo vw_documentos_status e gravando documento. Substituir mocks de documentos.ts/pendencias.ts (parte 'documento') pela API mantendo os tipos StatusPendencia do front.
  4. 4. Upload privado: Edge Functions documentos-upload-url e documentos-finalizar-upload; fluxo de câmera/PDF no front; validação de mime/tamanho client + server; limpeza de arquivo anterior.
  5. 5. Visualização segura: Edge Function documentos-download-url + visor in-app; gravação de evento 'visualizado'.
  6. 6. Push de marcos: registrar push_subscription (VAPID/Service Worker); cron diário + Edge Function de envio com idempotência via notificacao_log; marcos 30/15/7/1/0; remoção de subscriptions 410.
  7. 7. Agregação para score: RPC get_status_agregado_documentos (SECURITY DEFINER, GRANT restrito) e integração no portal B2B/seleção de carga 'validada' (selo documentação OK).
  8. 8. LGPD e hardening: Edge Function lgpd-excluir-documentos, rate-limit, get_advisors clean, testes (unit/integração/e2e) verdes, logs.

12Plano de teste

unit (função SQL rode_doc_status, mapeamento e agregação)

  • rode_doc_status: p_venc NULL -> 'sem_data' e dias_para_vencer NULL.
  • rode_doc_status com dias_alerta=30: hoje+40 -> 'em_dia', dias=40; hoje+15 -> 'proximo', dias=15; hoje-3 -> 'vencido', dias=-3; borda hoje+30 -> 'proximo'; borda hoje+31 -> 'em_dia'; borda hoje (d=0) -> 'proximo'.
  • rode_doc_status com dias_alerta=34 (RNTRC): hoje+34 -> 'proximo' (reconcilia mock pendencias.ts onde RNTRC a 34 dias é 'proximo').
  • Mapeamento StatusPendencia no front: 'em_dia'->'ok','proximo'->'proximo','vencido'->'vencido','sem_data'->'ok' sem faixa de alerta.
  • Ordenação DocumentoCard: vencido (dias=-5) acima de proximo (dias=3) acima de em_dia (dias=58) acima de sem_data; idêntico à lógica de PENDENCIAS_ORDENADAS.
  • Marco devido: dias_para_vencer=7 dispara marco 7; dias=6 não dispara nenhum; dias=0 dispara marco 0; dias=31 não dispara (fora de MARCOS).
  • Agregação rotulo: 1 doc vencido + resto ok -> 'vencido'; 0 vencidos + 1 proximo -> 'pendente'; 0 vencidos, 0 proximos, mas RNTRC obrigatório sem data -> 'pendente'; todos obrigatórios com data e em_dia -> 'em_dia'.
  • Zod EditarVencimento: rejeita data > hoje+3650 e data com tipo tem_vencimento=false; aceita data válida + numero opcional.

integração (RLS, RPC, Storage, Edge Functions, cron)

  • Motorista A não consegue SELECT/UPDATE/DELETE documento do Motorista B (RLS retorna 0 linhas; UPDATE afeta 0 rows).
  • INSERT em documento com motorista_id forjado (!= auth.uid()) é rejeitado pela policy WITH CHECK.
  • documentos-upload-url rejeita mime 'application/x-msdownload' (400) e tamanho 12MB (400); aceita image/jpeg 2MB e retorna arquivo_path com prefixo documentos/{uid}/.
  • documentos-finalizar-upload rejeita arquivo_path com prefixo de outro uid (403, anti-path-traversal) e arquivo_path inexistente no Storage (409); ao reenviar, o objeto anterior é apagado (bucket não acumula lixo).
  • documentos-download-url emite signed URL válida só para o dono e grava documento_evento 'visualizado'; chamada por outro motorista retorna 403 sem emitir URL; a URL expira após TTL 180s (GET após expiração -> 403/400 do Storage).
  • get_status_agregado_documentos retorna apenas {rotulo,total,vencidos,proximos,sem_data_obrigatorios,atualizado_em} e o JSON NÃO contém as chaves data_vencimento, numero nem arquivo_path; chamada por role authenticated genérica (sem grant) é negada.
  • upsert documento (mesmo motorista+tipo) atualiza a linha existente e grava documento_evento 'data_alterada' com payload {de,para}.
  • Idempotência do cron: rodar o cron 2x no mesmo dia gera apenas 1 registro UNIQUE(documento_id,marco,data_vencimento_ref) e 1 push por documento+marco; editar a data (novo data_vencimento_ref) permite novo marco sem violar UNIQUE.
  • push-subscribe faz upsert por endpoint (não duplica subscription do mesmo device); subscription com retorno 410 do envio é removida.
  • lgpd-excluir-documentos apaga todos os objetos sob documentos/{uid}/ e todas as linhas do motorista, grava 'excluido_lgpd', e get_status_agregado_documentos passa a retornar total=0.

e2e (PWA, fluxo do motorista e B2B)

  • Cadastrar vencimento da CNH para hoje+15: card passa a 'vence em 15 dias' (âmbar) e a faixa de aviso do Início mostra a CNH como próximo vencimento (PROXIMO_VENCIMENTO).
  • Editar data com optimistic update: UI atualiza imediatamente; ao simular erro 500 da API, há rollback visível e toast de erro, e o card volta ao estado anterior.
  • Enviar foto do CRLV (câmera): após upload card mostra 'arquivo enviado'; tocar 'Ver foto' abre a signed URL e o arquivo é exibido; reenviar substitui a foto (apenas 1 arquivo corrente).
  • Tentar enviar arquivo .exe ou de 12MB pela tela: bloqueado no cliente antes da chamada de rede, com mensagem de erro.
  • Documento vencido (RNTRC com data passada): card vermelho 'vencido há N dias'; o agregado vira 'vencido' e, no portal B2B, a seleção de carga 'validada' reflete documentação pendente (selo não-OK).
  • Tipo sem vencimento (CPF/CNPJ): card não mostra faixa de prazo nem botão 'Editar data', só upload/visualização.
  • Receber push: com vencimento em 7 dias e subscription ativa, após o cron o motorista recebe exatamente 1 notificação 'CNH vence em 7 dias' (mock do Service Worker valida payload).
  • Offline: com a rede desligada, a tela exibe a última lista cacheada; uma edição de data entra na fila e é sincronizada ao reconectar.

13Critérios de aceite

  • Dado um documento com data_vencimento = hoje+12 e tipo com dias_alerta=30, quando o motorista abre /documentos, então o card exibe status 'proximo' com 'vence em 12 dias' (âmbar) e aparece acima dos documentos 'em_dia' e abaixo dos 'vencido'.
  • Dado o RNTRC com tipo_documento.dias_alerta=34 e data_vencimento = hoje+34, quando vw_documentos_status é lida, então status='proximo' (reconciliando o mock pendencias.ts onde RNTRC a 34 dias é 'proximo').
  • Dado que o motorista edita a data de vencimento da CNH e salva, quando a próxima leitura de vw_documentos_status ocorre, então status e dias_para_vencer vêm recalculados sem nenhuma escrita manual de status, e exatamente 1 documento_evento 'data_alterada' com payload {de,para} é gravado.
  • Dado um tipo com tem_vencimento=false (CPF/CNPJ), quando o cliente tenta gravar data_vencimento, então a operação é rejeitada (400/CHECK) e o card não exibe campo nem faixa de data.
  • Dado um arquivo enviado, quando o motorista toca 'Ver foto', então o sistema gera signed URL com TTL <=180s, registra evento 'visualizado' e o arquivo abre; após o TTL a mesma URL deixa de funcionar (403/400 do Storage).
  • Dado um reenvio de arquivo no mesmo documento, quando finalizar-upload completa, então o objeto anterior é apagado do bucket e resta exatamente 1 arquivo corrente para o documento.
  • Dado o portal B2B consultando get_status_agregado_documentos, quando recebe a resposta, então ela contém apenas {rotulo,total,vencidos,proximos,sem_data_obrigatorios,atualizado_em} e NÃO contém data_vencimento, numero nem arquivo_path; e a RPC nega execução para role authenticated sem grant.
  • Dado um motorista com >=1 documento obrigatório vencido, quando o agregado é calculado, então rotulo='vencido'; dado 0 vencidos mas >=1 obrigatório sem data, então rotulo='pendente'; dado todos os obrigatórios com data e não-vencidos/não-proximos, então rotulo='em_dia'.
  • Dado um documento com vencimento em 7 dias e subscription push ativa, quando o cron diário roda, então exatamente 1 notificação do marco 7 é enviada e 1 linha UNIQUE(documento_id,7,data_vencimento_ref) é gravada em notificacao_log; reexecução no mesmo dia não gera novo envio.
  • Dado o Motorista A autenticado, quando tenta ler ou baixar documento do Motorista B, então a RLS/validação de ownership nega acesso (0 linhas / 403) e nenhuma signed URL é emitida.
  • Dado um upload, quando o mime não está em {jpeg,png,pdf} ou o tamanho > 10MB, então a Edge Function rejeita com 400 e nenhum objeto é criado no bucket.
  • Dado um pedido de exclusão LGPD confirmado, quando lgpd-excluir-documentos roda, então todos os objetos sob documentos/{uid}/ e todas as linhas de documento do motorista são removidos, um documento_evento 'excluido_lgpd' é gravado, e o agregado passa a retornar total=0.
