1Visão técnica

Evoluir o prototipo mockado da Calculadora de frete (canal App) para a killer feature real e offline-first do RODE COM LUCRO. O coracao ja existe e e puro/testavel: src/lib/calc.ts (calcularFrete(FreteInput):FreteResultado, FRETE_DEFAULTS, ROTAS, fmtBRL/fmtPct) e o UI src/components/ProfitCalculator.tsx + src/components/charts/CostBreakdown.tsx (donut de 7 fatias ja implementado, incluindo ARLA 32). O trabalho real NAO e o calculo, e: (1) formalizar a FORMULA DO EMERSON como modulo puro versionado (FORMULA_VERSAO='emerson-v1') e calibrado por casos reais; (2) trocar os presets ROTAS hardcoded por distancia+pedagio reais de uma API de roteirizacao de caminhao com cache (route_cache) e fallback determinista; (3) introduzir a TABELA DE PISO MINIMO ANTT versionada por vigencia e tipo de carga/eixos, derivando o veredito BOM/ACEITAVEL/RUIM por % acima do piso (estendendo a regra atual lucro<0/margem<meta); (4) persistir perfil do caminhao e historico de analises com sync offline-first idempotente (UUID gerado no cliente, upsert merge-duplicates); (5) preco de diesel por regiao como input default editavel. Arquitetura: PWA React+TS (Vite, Workbox) mobile-first com motor de calculo 100% no cliente (deterministico, mesmo bundle compartilhado client/server via @rode/calc), Supabase Postgres com RLS por usuario, Edge Functions (Deno/TS) para roteirizacao, piso ANTT, diesel e exclusao LGPD. O calculo e a fonte de verdade local; o backend so fornece insumos (distancia, pedagio, piso, diesel) e persiste resultado com snapshots e versoes para reprodutibilidade. O veredito financeiro e estimativa com disclaimer obrigatorio (sensibilidade juridica do piso ANTT). Principio: o motorista parado no posto, com 4G ruim ou offline, obtem veredito em <2s usando o ultimo insumo em cache + presets, e o Salvar nunca bloqueia esperando rede.

2Arquitetura

  • Motor de calculo puro e isomorfico ja existente em src/lib/calc.ts: calcularFrete(input:FreteInput):FreteResultado sem efeitos colaterais, sem fetch, recebendo TODOS os insumos como argumentos (distanciaKm, valorFrete, pedagio, precoDiesel, consumoKmL, custos/km, dias, voltaVazia, margemDesejadaPct). Extrair para pacote compartilhavel @rode/calc importado pelo PWA, pela Edge Function do feed (find-app) e pelo copiloto WhatsApp (calc-wpp) — uma unica fonte de verdade da formula, nunca reimplementada no backend. Adicionar export const FORMULA_VERSAO = 'emerson-v1' e carimba-lo em cada analise persistida.
  • Offline-first como invariante de produto: o calculo roda 100% no cliente (determinismo, <16ms via useMemo). Insumos externos (distancia/pedagio por rota, piso ANTT por vigencia, diesel por regiao) sao buscados via Edge Function mas cacheados em IndexedDB (idb) e no Cache API do Service Worker (Workbox, stale-while-revalidate). Sem rede usa-se o ultimo insumo conhecido + presets ROTAS (ja em calc.ts) por aproximacao de cidade, sinalizando 'estimativa offline'. O resultado NUNCA depende de round-trip de servidor.
  • BFF fino em Supabase Edge Functions (Deno/TS) so para o que o cliente nao pode fazer com seguranca/eficiencia: (a) /route-cost proxy+cache da API de roteirizacao truck (esconde API key, normaliza, cacheia por origem-destino+eixos); (b) /antt-piso serve tabela ANTT versionada; (c) /diesel-regiao serve preco de referencia; (d) /apagar-conta para LGPD. Sync de perfil/historico via PostgREST com RLS. O LLM/NLU NAO participa deste modulo (e do canal WhatsApp).
  • Persistencia em Supabase Postgres 15 com Row Level Security: cada motorista so le/escreve suas linhas (caminhao_perfil, analise_frete). Escrita offline com id uuid gerado no cliente e fila de sync idempotente (upsert por id, Prefer: resolution=merge-duplicates) para nunca duplicar quando o sinal voltar.
  • Versionamento de regras como DADOS, nao codigo: antt_piso_tabela e cost_defaults ficam no banco com colunas de vigencia (valido_de/valido_ate). Cada analise persiste a versao usada (antt_versao, formula_versao) e os snapshots congelados (insumos_snapshot, resultado_snapshot) para que o historico permaneca reproduzivel mesmo apos atualizacao das tabelas de referencia.
  • Camada de roteirizacao com adaptador plugavel: interface RouteProvider { resolve(origem,destino,eixos,tipoCarga): Promise<{distanciaKm,pedagioCentavos,provider}> }. Implementacao inicial com 1 provedor (OpenRouteService driving-hgv | Google Routes | HERE) atras de /route-cost. Cascata de fallback determinada e testavel: API -> route_cache do par -> preset ROTAS por match de cidade -> entrada manual do motorista (sempre editavel, marca origem_dado='manual').
  • Fronteira de confianca e disclaimer: o veredito financeiro e ESTIMATIVA. O piso ANTT e referencia regulatoria sensivel; toda tela Resultado exibe disclaimer obrigatorio e nao bloqueia o motorista. O backend so fornece insumos e persiste; a fonte de verdade do calculo e local.

4Modelo de dados

  • apelido text
  • categoria text CHECK in ('truck','carreta_simples','carreta_6_eixos','bitrem','rodotrem')
  • eixos int CHECK (eixos BETWEEN 2 AND 9)
  • consumo_km_l numeric(4,2) NOT NULL DEFAULT 2.5 CHECK (>0)
  • preco_diesel numeric(6,3) CHECK (>=0)
  • arla_pct numeric(4,2) NOT NULL DEFAULT 5 CHECK (BETWEEN 0 AND 100)
  • manutencao_km numeric(6,3) NOT NULL DEFAULT 0.35
  • pneus_km numeric(6,3) NOT NULL DEFAULT 0.12
  • depreciacao_km numeric(6,3) NOT NULL DEFAULT 0.25
  • alimentacao_dia numeric(8,2) NOT NULL DEFAULT 90
  • margem_desejada_pct numeric(4,1) NOT NULL DEFAULT 20 CHECK (BETWEEN 0 AND 100)
  • capacidade_tanque_l int CHECK (>0)
  • uf_base text CHECK (char_length(uf_base)=2)
  • created_at timestamptz DEFAULT now()
  • updated_at timestamptz DEFAULT now(). Indice (user_id).

  • pct_acima_piso numeric(6,2)
  • piso_antt_centavos bigint

5Contratos de API

GET/functions/v1/route-cost?origem={cidade-uf}&destino={cidade-uf}&eixos={n}&tipoCarga={t}

Edge Function (Deno). Resolve distancia (km, so ida) e pedagio (centavos) por rota de caminhao. Cascata: route_cache (fresco por ttl_dias) -> provedor externo perfil HGV/truck -> resposta com fallback=true. 200 {distanciaKm:number, pedagioCentavos:number, provider:string, fonte:'cache'|'api', coletadoEm:ISO8601, eixos:number}. Quando provedor falha ou origem/destino nao geocodificam: 200 {fallback:true, motivo:'provider_down'|'geocode_miss'|'rate_limit'} (NUNCA 5xx que quebre o cliente) para o cliente cair em presets ROTAS. Esconde API key (secret da function). Rate-limit 30 req/min/usuario via token bucket; ao estourar retorna {fallback:true, motivo:'rate_limit'}. Normaliza origem/destino (lower, sem acento, trim) antes do cache lookup.

GET/functions/v1/antt-piso?tipoCarga={t}&eixos={n}&data={yyyy-mm-dd}

Retorna o coeficiente do piso minimo ANTT vigente na data informada (default hoje) para tipoCarga e eixos. 200 {coefDeslocamentoCentavos:number, coefCargaDescargaCentavos:number, resolucao:string, anttVersao:string, validoDe:date, validoAte:date|null, fonteUrl:string}. 404 {erro:'sem_vigencia'} quando nao ha linha vigente para a combinacao. O cliente calcula piso=coefDeslocamento*distanciaKm(ida)+coefCargaDescarga e cacheia a tabela inteira (todas as combinacoes vigentes) em IndexedDB para uso offline. Resposta com Cache-Control para stale-while-revalidate no SW.

GET/functions/v1/diesel-regiao?uf={uf}&rodovia={br}

Retorna preco de diesel de referencia por regiao para pre-preencher precoDiesel (sempre editavel). 200 {precoS10Centavos:number, precoS500Centavos:number, uf:string, regiao:string, coletadoEm:ISO8601, fonte:string}. Fallback para media nacional quando nao ha dado regional. Cacheavel offline (SW stale-while-revalidate, TTL 24h).

GET/rest/v1/caminhao_perfil?user_id=eq.{uid}

PostgREST com RLS (user_id=auth.uid()). Le perfil(is) do caminhao do motorista (consumo, custos/km, margem, diesel, tanque). Usado para preencher defaults da analise. Sem JWT valido retorna 401; perfil de outro usuario retorna [] (RLS).

POST/PATCH/rest/v1/caminhao_perfil (Prefer: resolution=merge-duplicates)

Cria/atualiza perfil do caminhao (upsert por id uuid gerado no cliente). Body: apelido, categoria, eixos, consumo_km_l, preco_diesel, arla_pct, manutencao_km, pneus_km, depreciacao_km, alimentacao_dia, margem_desejada_pct, capacidade_tanque_l, uf_base. RLS forca user_id=auth.uid() via policy WITH CHECK; tentativa de gravar com user_id de terceiro e rejeitada (403).

POST/rest/v1/analise_frete (Prefer: resolution=merge-duplicates,return=minimal)

Persiste analise com id uuid gerado no cliente (idempotente). Body: caminhao_perfil_id, origem_cidade/uf, destino_cidade/uf, distancia_km, volta_vazia, valor_frete_centavos, pedagio_centavos, dias, insumos_snapshot(jsonb), resultado_snapshot(jsonb), veredito, pct_acima_piso, piso_antt_centavos, formula_versao, antt_versao, route_provider, origem_dado. Reenvio com mesmo id faz upsert (merge-duplicates) sem duplicar. Usado pela fila de sync offline. RLS WITH CHECK user_id=auth.uid().

GET/rest/v1/analise_frete?user_id=eq.{uid}&order=created_at.desc&limit=50&offset={n}

Lista historico de analises do motorista (paginado, 50/pagina) para a tela Historico. RLS restringe ao proprio usuario. Suporta filtro select= para trazer so colunas da lista (origem/destino/valor/veredito/created_at) e adiar os snapshots jsonb.

POST/functions/v1/apagar-conta

Edge Function autenticada. Direito de exclusao LGPD: apaga analise_frete e caminhao_perfil do auth.uid() (cascade) e remove o usuario do Auth. 200 {removidos:{analises:n, perfis:n}}. Idempotente. Registra evento de auditoria sem PII (apenas user_id hash + timestamp).

7Lógica de negócio

  • FORMULA DO EMERSON (custo por viagem) ja em calcularFrete() — pura: fatorDistancia = voltaVazia ? 2 : 1; distanciaTotalKm = distanciaKm*fatorDistancia; litros = distanciaTotalKm/consumoKmL; custoDiesel = litros*precoDiesel; custoArla = custoDiesel*(arlaPct/100); custoManutencao = distanciaTotalKm*manutencaoKm; custoPneus = distanciaTotalKm*pneusKm; custoDepreciacao = distanciaTotalKm*depreciacaoKm; custoPedagio = pedagio*fatorDistancia; custoAlimentacao = alimentacaoDia*dias; custoTotal = soma. lucro = valorFrete-custoTotal; margemRealPct = valorFrete>0 ? lucro/valorFrete*100 : 0; valorMinimo = custoTotal*(1+margemDesejadaPct/100); custoPorKm = distanciaTotalKm>0 ? custoTotal/distanciaTotalKm : 0; receitaPorKm = distanciaKm>0 ? valorFrete/distanciaKm : 0. Carimbar FORMULA_VERSAO='emerson-v1'; calibrar coeficientes com 3-5 casos reais do Emerson antes do go-live.
  • PISO MINIMO ANTT (calculado no cliente com insumo de /antt-piso): piso = coefDeslocamento(tipoCarga,eixos)*distanciaKm(ida) + coefCargaDescarga(tipoCarga,eixos). pctAcimaPiso = piso>0 ? (valorFrete-piso)/piso*100 : 0. Insumo versionado por vigencia (valido_de/valido_ate); congelar antt_versao na analise. Em centavos no transporte; conversao a reais so na UI.
  • VEREDITO (estende a regra atual lucro<0->ruim; margem<meta->aceitavel; else bom com a checagem de piso): SE lucro<0 OU valorFrete<piso -> 'ruim'; SENAO SE margemRealPct<margemDesejadaPct OU pctAcimaPiso<LIMIAR_VERDE (parametrizavel, ex.: 15%) -> 'aceitavel'; SENAO -> 'bom'. LIMIAR_VERDE/amarelo calibrados empiricamente com o Emerson. Quando o piso nao esta disponivel (offline sem cache) o veredito cai na regra base sem piso e a UI sinaliza 'piso indisponivel'.
  • RESOLUCAO DE ROTA: normalizar origem/destino (cidade+UF, minusculas, sem acento, trim) -> route_cache (se fresco por ttl_dias) -> API truck (perfil HGV) -> preset ROTAS por match de cidade -> entrada manual. Pedagio na volta: se voltaVazia, custoPedagio = pedagio*2 (caminhao volta dirigindo) — ja em calc.ts. Dias estimados default por faixa de km (<=600:1; <=1200:2; <=2000:3; <=3000:4; >3000:5) e editavel; presets ROTAS ja trazem dias por rota.
  • SYNC/IDEMPOTENCIA: id da analise e UUID gerado no cliente; persistencia local imediata em IndexedDB (synced=false); POST com Prefer resolution=merge-duplicates faz upsert por id; ao 2xx marca synced=true. Retry da fila com backoff exponencial nao duplica (upsert por id). Campos monetarios em centavos inteiros no banco; conversao a reais so na UI (fmtBRL/fmtBRLCent ja existem).
  • FORMATACAO/UX: fmtBRL, fmtBRLCent, fmtPct, fmtNum ja implementados (Intl pt-BR). Explicacao textual do veredito por explicar(resultado,margem) (ja existe) — estender para mencionar o piso ANTT quando valorFrete<piso ('abaixo do piso minimo da ANTT em X%'). Tratamento de divisao por zero ja coberto em calc.ts (valorFrete=0 e distanciaKm=0).

9Segurança & LGPD

  • RLS no Postgres: policies em caminhao_perfil e analise_frete com USING (user_id=auth.uid()) para SELECT/UPDATE/DELETE e WITH CHECK (user_id=auth.uid()) para INSERT/UPDATE. Nenhum motorista acessa ou grava dados de outro. Tabelas de referencia (antt_piso_tabela, cost_defaults, route_cache, diesel_regiao) sao read-only para 'authenticated'; escrita so via service role (Edge Functions/jobs).
  • LGPD minimizacao: a analise guarda origem/destino e valores que o proprio motorista digitou — sem dado sensivel de terceiros. Consentimento explicito no onboarding para salvar historico e perfil; sem consentimento o app funciona em modo so-calculo (nao persiste). Direito de exclusao via Edge Function /apagar-conta (remove analise_frete e caminhao_perfil em cascade + usuario do Auth, idempotente).
  • LGPD agregacao anonimizada: qualquer uso analitico (admin/B2B) das analises e AGREGADO (media de margem por rota, distribuicao de veredito) sem expor o individuo, alinhado ao Painel administrativo. Rota/financeiro individual nunca e exposto a B2B ou admin identificavel.
  • Disclaimer juridico obrigatorio no Resultado (DisclaimerAntt): 'Estimativa com base em modelo de custo transparente; o piso ANTT e referencia regulatoria e o veredito nao e aconselhamento.' Mitiga a sensibilidade do piso (risco listado no PRD); o disclaimer e renderizado sempre que houver linha de piso.
  • Segredos e abuso: API key do provedor de roteirizacao em secret da Edge Function, nunca no bundle do PWA. /route-cost com rate-limit 30 req/min/usuario (token bucket) para conter custo/abuso; ao estourar retorna fallback (presets) em vez de erro. Diesel e piso sao publicos (nao sensiveis). Validacao de input nas Edge Functions (origem/destino sanitizados, eixos 2-9, tipoCarga em enum) para prevenir injection no provedor.
  • Auth por telefone OTP (Supabase Auth) com sessao; tokens em storage do PWA gerido pelo supabase-js. PostgREST so aceita requests autenticados (JWT) para tabelas de usuario; /apagar-conta exige JWT valido e opera sobre auth.uid() (nunca aceita user_id no body).

10Não-funcionais

  • Performance: veredito recalculado no cliente em <16ms (puro, sem rede, validado por benchmark Vitest); tela Resultado renderiza em <2s mesmo em 3G; resolucao de rota com debounce 400ms e cache para nunca bloquear o calculo. LCP do shell <2.5s em rede 4G de posto.
  • Offline-first: calculo 100% local; sem rede usa ultimo insumo em cache + presets ROTAS com aviso 'estimativa offline'. Salvar funciona offline (IndexedDB) e sincroniza sem duplicar (upsert por id) quando o sinal voltar.
  • Confiabilidade/determinismo: calcularFrete() puro e coberto por testes unitarios; mesmos inputs -> mesmo output. A analise persiste insumos_snapshot e resultado_snapshot + versoes (formula_versao, antt_versao) para reprodutibilidade do historico mesmo apos atualizacao das tabelas de referencia.
  • Versionamento sem quebrar historico: atualizar antt_piso_tabela ou cost_defaults cria nova vigencia (valido_de/valido_ate); analises antigas mantem a versao que usaram. Migracoes de schema reversiveis (up/down).
  • Precisao/calibracao: erro do veredito controlado calibrando defaults por categoria com casos reais do Emerson; aviso explicito de que rota/pedagio sao estimados e editaveis; meta de erro <10% entre distancia estimada e ajuste manual medido em producao.
  • Acessibilidade e moeda: valores em centavos inteiros no banco, conversao a reais so na UI (locale pt-BR); inputmode='decimal' nos campos numericos; toggle volta-vazia com role=switch e aria-checked (ja no mock); contraste AA nos chips de veredito.

12Plano de teste

Unit (Vitest) — motor de calculo puro src/lib/calc.ts

  • calcularFrete com FRETE_DEFAULTS + rota SP->Recife (distanciaKm=2650, valorFrete=8000, pedagio=720, voltaVazia=false): litros=2650/2.5=1060; custoDiesel=1060*6.1=6466.00; custoArla=6466*0.05=323.30; custoManutencao=2650*0.35=927.50; custoPneus=2650*0.12=318.00; custoDepreciacao=2650*0.25=662.50; custoPedagio=720; custoAlimentacao=90*4=360; custoTotal soma=9777.30; lucro=8000-9777.30=-1777.30; veredito='ruim'. Asserir cada componente com tolerancia 0.01.
  • voltaVazia=true dobra distanciaTotalKm e custoPedagio: distanciaTotalKm=distanciaKm*2 e custoPedagio=pedagio*2; custoAlimentacao NAO dobra (depende de dias, nao de distancia); receitaPorKm continua sobre distanciaKm (ida) e nao sobre o total.
  • veredito de fronteira: lucro<0 -> 'ruim'; 0<=margemRealPct<margemDesejadaPct -> 'aceitavel'; margemRealPct>=margemDesejadaPct -> 'bom'. Caso exato margemRealPct==margemDesejadaPct deve ser 'bom' (>=).
  • piso ANTT: dado coefDeslocamento e coefCargaDescarga p/ carga_geral/eixos, piso=coef*distanciaKm(ida)+cargaDescarga; quando valorFrete<piso o veredito e forcado a 'ruim' mesmo com lucro>0; pctAcimaPiso=(valorFrete-piso)/piso*100 com sinal negativo quando abaixo.
  • bordas numericas: valorFrete=0 -> margemRealPct=0 e receitaPorKm=0 sem divisao por zero; distanciaKm=0 -> custoPorKm=0 e receitaPorKm=0; consumoKmL>0 sempre (input validado).
  • arlaPct aplicado SO sobre custoDiesel (custoArla=custoDiesel*arlaPct/100); alterar arlaPct nao afeta manutencao/pneus/depreciacao.
  • determinismo/benchmark: 10.000 execucoes de calcularFrete com inputs identicos retornam exatamente o mesmo FreteResultado; tempo medio por chamada <0.1ms (garante o orcamento <16ms da UI).

Unit (Vitest) — adaptador de rota e cascata de fallback

  • normalizacao: 'Sao Paulo (SP)' e 'sao paulo, sp' produzem o mesmo origem_norm ('sao paulo|sp'); acentos e maiusculas removidos.
  • cascata: provider resolve -> usa API; provider lanca -> usa route_cache se fresco; cache expirado por ttl_dias -> tenta API de novo; tudo indisponivel -> retorna preset ROTAS por match de cidade; sem preset -> origem_dado='manual' aguardando input.
  • faixa de dias por km: 600->1, 601->2, 1200->2, 1201->3, 2000->3, 3000->4, 3001->5.

Integracao (Vitest + Supabase local / mocks das Edge Functions)

  • GET /route-cost retorna fonte='cache' quando ha linha fresca em route_cache e NAO chama o provedor externo (spy no provider nao invocado); quando coletado_em+ttl_dias expirou, chama o provedor e faz upsert no cache.
  • GET /route-cost com provedor indisponivel retorna 200 {fallback:true,motivo:'provider_down'} (nunca 5xx) e o cliente cai nos presets ROTAS; ao estourar rate-limit retorna {fallback:true,motivo:'rate_limit'}.
  • GET /antt-piso?data=2024-06-01 vs ?data=2025-06-01 retorna a vigencia correta quando ha duas versoes (valido_de/valido_ate diferentes); data sem vigencia retorna 404 {erro:'sem_vigencia'}.
  • POST analise_frete com o MESMO id duas vezes (Prefer merge-duplicates) resulta em 1 linha (count=1) — idempotencia da fila de sync; segundo POST nao altera created_at original.
  • POST analise_frete persiste insumos_snapshot, resultado_snapshot, veredito, pct_acima_piso, piso_antt_centavos, formula_versao e antt_versao exatamente como enviados; valores monetarios em centavos inteiros.
  • /apagar-conta remove todas as analise_frete e caminhao_perfil do auth.uid() e e idempotente (2a chamada retorna removidos:0 sem erro).

Integracao — RLS (Supabase local, pgTAP/SQL com 2 JWTs)

  • motorista A (jwt A) faz SELECT em analise_frete do motorista B -> retorna 0 linhas; SELECT em caminhao_perfil de B -> 0 linhas.
  • motorista A tenta INSERT em analise_frete com user_id=B no body -> rejeitado pela policy WITH CHECK (403/erro), nenhuma linha criada.
  • motorista A tenta UPDATE/DELETE de linha de B -> 0 linhas afetadas.
  • usuario anonimo (sem JWT) faz GET /rest/v1/caminhao_perfil -> 401; tabelas de referencia (antt_piso_tabela, diesel_regiao) sao legiveis por authenticated mas nao graváveis por authenticated (INSERT negado).

13Critérios de aceite

  • DADO um frete SP->Recife de R$ 8.000 com os defaults do Emerson (consumo 2,5 km/L, diesel R$ 6,10, distancia 2650 km, pedagio R$ 720, 4 dias, volta-vazia=nao), QUANDO o motorista abre o Resultado, ENTAO o app exibe custo estimado de R$ 9.777,30, prejuizo provavel de R$ 1.777,30, margem real negativa, 'Negocie a partir de' (custoTotal*1,20) e o veredito RUIM em ate 2s.
  • DADO que o motorista mexe em qualquer campo (margem, valor, volta-vazia, diesel, custos/km), QUANDO altera o valor, ENTAO o veredito e os KPIs recalculam no cliente em <16ms, sem qualquer chamada ao servidor (verificavel por ausencia de request de rede).
  • DADO um frete cujo valor esta abaixo do piso minimo ANTT vigente para o tipo de carga e numero de eixos, QUANDO o Resultado e exibido, ENTAO o veredito e RUIM, a PisoAnttRow mostra o '% abaixo do piso' e o DisclaimerAntt ('o piso ANTT e referencia regulatoria; o veredito nao e aconselhamento') esta visivel.
  • DADO um frete com lucro positivo cuja margemRealPct e maior ou igual a margemDesejadaPct E acima do piso ANTT, QUANDO o Resultado e exibido, ENTAO o veredito e BOM; SE margemRealPct < margemDesejadaPct (com lucro>=0 e acima do piso) ENTAO o veredito e ACEITAVEL.
  • DADO que o app esta offline, QUANDO o motorista analisa um frete, ENTAO o calculo roda local usando o ultimo insumo em cache ou preset ROTAS, exibe o OfflineBanner 'estimativa offline' e permite Salvar; ao reconectar, a analise sincroniza (upsert por id) sem duplicar e o badge 'pendente' some.
  • DADO que o motorista salva uma analise, QUANDO ela e persistida, ENTAO o registro guarda insumos_snapshot, resultado_snapshot, veredito, pct_acima_piso, piso_antt_centavos, formula_versao='emerson-v1' e antt_versao, com valores monetarios em centavos, e aparece na tela Historico ordenado por created_at desc.
  • DADO que a tabela antt_piso_tabela e atualizada com uma nova vigencia, QUANDO o motorista reabre uma analise antiga no Historico, ENTAO ela mantem os valores e a antt_versao usados na epoca, sem recalcular com a tabela nova.
  • DADO um par origem-destino+eixos ja consultado dentro do TTL (ttl_dias), QUANDO o motorista repete a rota, ENTAO a distancia/pedagio vem do route_cache (fonte='cache') sem nova chamada paga ao provedor externo.
  • DADO que o motorista A esta autenticado, QUANDO tenta ler ou gravar analise_frete ou caminhao_perfil do motorista B, ENTAO a RLS impede (0 linhas no SELECT, 403 no INSERT com user_id de terceiro) e nenhum dado de B e retornado.
  • DADO que o motorista solicita 'apagar minha conta', QUANDO a Edge Function /apagar-conta executa, ENTAO todas as suas analise_frete e caminhao_perfil sao removidas em cascade e a operacao e idempotente (2a chamada nao falha).
