1Visão técnica

Módulo 'Diesel na rota' no canal WhatsApp do RODE COM LUCRO: serviço proativo, event-driven e serverless (Supabase Edge Functions + Postgres/PostGIS) que infere a posição aproximada do caminhoneiro (compartilhamento de localização via WhatsApp, última cidade/rota conhecida do app ou check-in manual), cruza com base de postos georreferenciada cujos preços vêm do módulo de coleta (OCR/crowdsourcing — dependência de dados, não de runtime) e, quando um posto À FRENTE está abaixo da média da rota por um limiar relevante (>=3% OU >=R$0,15/L), dentro de raio configurável (default 8 km) e do TTL do preço (default 12 h), dispara um template HSM aprovado pela Meta com a economia personalizada pela capacidade do tanque (ex.: 700 L × R$0,27/L = ~R$190). Três preocupações inegociáveis governam o design: (1) CONFIANÇA — nunca alertar com preço velho nem posto no sentido contrário; (2) CUSTO — dedupe, teto diário, janela de silêncio (America/Sao_Paulo) e kill-switch global ANTES de gastar uma conversa HSM paga; (3) LGPD — geolocalização é dado pessoal: opt-in explícito com base legal, retenção mínima (driver_location expira em 6 h), opt-out via PARAR honrado em <5 s, RLS por driver_id. O módulo NÃO coleta preço e NÃO faz navegação turn-by-turn (apenas link 'ver no mapa'). Reaproveita os parâmetros do mock em src/data/diesel.ts (Ourinhos/BR-153 km318, média R$6,16, Posto X R$5,89, delta R$0,27, 700 L, ~R$190, 6 km à frente, posição atual km312) como UI e fixtures de teste, e fecha o ciclo com feedback loop (entrega/leitura/clique/abastecimento) para calibrar o limiar e medir economia real e custo por alerta útil. Estimativa de referência (PRD de produto diesel-wpp): 7–12 dev-dias, 2–3 semanas, assumindo base de preços georreferenciada e canal WhatsApp já existentes.

2Arquitetura

  • Pipeline event-driven serverless: ingestão de posição (diesel-location-ingest) -> avaliação de gatilho (diesel-alert-evaluate) -> outbox (dispatch_queue) -> worker de envio (wpp-send-hsm) -> webhook de status (wpp-webhook) -> feedback loop (alert_feedback). Cada etapa é uma Edge Function isolada, idempotente e reexecutável; um pico de posições não derruba o canal WhatsApp porque o envio é desacoplado por fila com SKIP LOCKED.
  • Geoespacial no Postgres com PostGIS: postos e localização como geometry(Point,4326); candidatos por ST_DWithin(geom_motorista, geom_posto, raio_km*1000); filtro de direção pelo ângulo entre heading do motorista e o bearing motorista->posto via ST_Azimuth (aprova se diferença angular normalizada <=90°). Evita sugerir posto no sentido contrário — risco crítico citado no PRD de produto. Coordenadas em SRID 4326; distâncias calculadas com geography (ST_DistanceSphere/ST_DWithin sobre ::geography) para precisão em metros.
  • Separação de responsabilidades: este módulo NÃO coleta preço (OCR/crowdsourcing é o módulo diesel-app) — apenas consome a view preco_vigente já validada. Acoplamento por dados (tabela preco_posto + TTL), nunca por chamada síncrona; se o módulo de coleta cair, o gatilho degrada para 'sem dado fresco' e fica silencioso.
  • Controle de custo de primeira classe: ANTES de gastar uma conversa HSM paga, todo disparo passa por dedupe (driver_id+posto_id+janela de hora), teto diário por motorista (default 4), frequência mínima entre alertas (default 60 min), janela de silêncio (22:00–06:00 America/Sao_Paulo) e kill-switch global + teto de custo agregado do dia.
  • WhatsApp Cloud API (Meta Graph API v20+) como canal único: templates HSM pré-aprovados para mensagem proativa fora da janela de 24 h; webhook único para status (sent/delivered/read/failed) e respostas de botão (PARAR/AJUSTAR/VER NO MAPA). BSP (Twilio/Gupshup/Zenvia) como fallback de portabilidade atrás da interface WppGateway, para não acoplar o domínio ao provedor.
  • RLS (Row Level Security) por driver_id em todas as tabelas com PII (driver_location, diesel_alert_pref, diesel_alert, consent_log, alert_feedback). Edge Functions usam service_role apenas na camada de aplicação após resolver o dono; o exterior nunca recebe SQL livre. O painel admin lê apenas agregados (materialized view), sem lat/lng identificável.
  • Observabilidade ponta-a-ponta: alert_id propaga por evaluate -> dispatch_queue -> wpp-send-hsm -> wpp_message_log -> alert_feedback, permitindo medir latência (p95), CTR, economia gerada e custo por alerta útil. Todo skip carrega skip_reason tipado para diagnóstico do funil.
  • Resiliência: falha do provedor de map-matching degrada para PostGIS local (bearing/distância) sem derrubar o gatilho; falha da Cloud API mantém itens em 'queued' sem perda; reentrega de webhook é idempotente por messages[].id / statuses[].id.

4Modelo de dados

diesel_alert_pref

  • driver_id uuid PK FK->driver, opt_in boolean NOT NULL default false, opt_in_at timestamptz null, opt_out_at timestamptz null, raio_km int NOT NULL default 8 CHECK (raio_km BETWEEN 5 AND 50), teto_dia int NOT NULL default 4 CHECK (teto_dia BETWEEN 1 AND 20), frequencia_min int NOT NULL default 60 CHECK (frequencia_min BETWEEN 15 AND 240), janela_silencio_inicio time NOT NULL default '22:00', janela_silencio_fim time NOT NULL default '06:00', limiar_pct numeric(4,2) NOT NULL default 3.0, limiar_litro_centavos int NOT NULL default 15, consent_location boolean NOT NULL default false, updated_at timestamptz NOT NULL default now() — RLS: dono apenas

driver_location

  • id uuid PK default gen_random_uuid(), driver_id uuid NOT NULL FK->driver, geom geometry(Point,4326) NOT NULL, heading_deg numeric(5,2) null, speed_kmh numeric(5,1) null, accuracy_m numeric(7,1) null, source text NOT NULL CHECK (source IN ('live_share','last_route','checkin')), created_at timestamptz NOT NULL default now(), expires_at timestamptz NOT NULL default now()+interval '6 hours' — índice GiST(geom)
  • índice (driver_id, created_at desc)
  • job de purga por expires_at
  • RLS por driver_id

posto

  • id uuid PK default gen_random_uuid(), nome text NOT NULL, rodovia text, km int, cidade text, uf char(2), geom geometry(Point,4326) NOT NULL, lado_pista text CHECK (lado_pista IN ('crescente','decrescente','ambos')) default 'ambos', ativo boolean NOT NULL default true — índice GiST(geom)
  • seed: Posto X (BR-153 km318 Ourinhos/SP)

preco_posto

  • id uuid PK default gen_random_uuid(), posto_id uuid NOT NULL FK->posto, tipo text NOT NULL CHECK (tipo IN ('S10','S500')), preco_centavos int NOT NULL CHECK (preco_centavos > 0), fonte text, confirmacoes int NOT NULL default 1, coletado_at timestamptz NOT NULL default now(), ttl_min int NOT NULL default 720, estado text NOT NULL CHECK (estado IN ('pendente','confirmado','suspeito')) default 'pendente' — view preco_vigente = registros onde estado='confirmado' AND now() < coletado_at + (ttl_min||' min')::interval
  • índice (posto_id, tipo, coletado_at desc)

media_rota

  • id uuid PK default gen_random_uuid(), rodovia text NOT NULL, cidade text, uf char(2), tipo text NOT NULL CHECK (tipo IN ('S10','S500')), media_centavos int NOT NULL, n_postos int NOT NULL CHECK (n_postos >= 1), calculado_at timestamptz NOT NULL default now() — agregada por pg_cron a partir de preco_vigente
  • usada como referência do limiar
  • índice (rodovia, cidade, tipo)

diesel_alert

  • id uuid PK default gen_random_uuid(), driver_id uuid NOT NULL FK, posto_id uuid FK null, tipo text CHECK (tipo IN ('S10','S500')), preco_centavos int null, media_centavos int null, delta_litro_centavos int null, pct_abaixo numeric(5,2) null, litros_tanque int null, economia_centavos int null, distancia_km numeric(6,2) null, decision text NOT NULL CHECK (decision IN ('fired','skipped')), skip_reason text null CHECK (skip_reason IN ('preco_velho','sem_candidato','sentido_contrario','abaixo_limiar','dedupe','teto_dia','frequencia','janela_silencio','opt_out','kill_switch','custo_teto')), dedupe_key text UNIQUE, created_at timestamptz NOT NULL default now() — RLS por driver_id
  • índice (driver_id, created_at desc)

dispatch_queue

  • id uuid PK default gen_random_uuid(), alert_id uuid NOT NULL UNIQUE FK->diesel_alert, driver_id uuid NOT NULL, driver_phone text NOT NULL, template text NOT NULL, vars jsonb NOT NULL, buttons jsonb, status text NOT NULL CHECK (status IN ('queued','sending','sent','delivered','read','failed','canceled')) default 'queued', wamid text null, attempts int NOT NULL default 0, next_attempt_at timestamptz null, locked_at timestamptz null, custo_centavos int null, last_error text null, created_at timestamptz NOT NULL default now(), updated_at timestamptz NOT NULL default now() — outbox SKIP LOCKED
  • UNIQUE(alert_id) garante envio único
  • índice parcial (status,next_attempt_at) WHERE status IN ('queued','sending')

wpp_message_log

  • id uuid PK default gen_random_uuid(), wamid text, provider_msg_id text, direction text NOT NULL CHECK (direction IN ('out','in')), payload jsonb NOT NULL, status text, error_code text null, driver_id uuid null, created_at timestamptz NOT NULL default now() — trilha completa Meta
  • UNIQUE(direction, provider_msg_id) para idempotência de webhook
  • índice (wamid), (driver_id, created_at desc)

consent_log

  • id uuid PK default gen_random_uuid(), driver_id uuid NOT NULL FK, finalidade text NOT NULL CHECK (finalidade IN ('location','wpp_alert')), acao text NOT NULL CHECK (acao IN ('opt_in','opt_out')), canal text NOT NULL CHECK (canal IN ('app','whatsapp','admin')), base_legal text NOT NULL default 'consentimento', texto_versao text, created_at timestamptz NOT NULL default now() — LGPD auditável, APPEND-ONLY (trigger bloqueia UPDATE/DELETE)
  • índice (driver_id, finalidade, created_at desc)

alert_feedback

  • id uuid PK default gen_random_uuid(), alert_id uuid NOT NULL FK->diesel_alert, evento text NOT NULL CHECK (evento IN ('delivered','read','clicked','abastecido')), valor_centavos int null, created_at timestamptz NOT NULL default now() — UNIQUE(alert_id, evento) para idempotência
  • calibra limiar e mede economia/CTR
  • índice (alert_id)

system_flag

  • key text PK, bool_value boolean, int_value int, updated_by text, updated_at timestamptz NOT NULL default now() — guarda kill_switch_global (bool) e custo_teto_dia_centavos (int)
  • lido antes de cada envio

5Contratos de API

GET/POST/functions/v1/wpp-webhook

Webhook único da Meta Cloud API. GET (verificação): valida hub.mode=subscribe e hub.verify_token==VERIFY_TOKEN -> responde hub.challenge com 200; token errado -> 403. POST: valida assinatura X-Hub-Signature-256 (HMAC-SHA256 do corpo cru com APP_SECRET) ANTES de qualquer parse -> assinatura inválida = 401 sem processamento. Recebe statuses[] (sent/delivered/read/failed) e messages[] (texto livre e quick-reply PARAR/AJUSTAR/VOLTAR/VER NO MAPA). Idempotente por statuses[].id e messages[].id (UNIQUE em wpp_message_log). Responde SEMPRE 200 em <5 s; o processamento real é enfileirado. Roteamento: PARAR->opt-out imediato + cancela dispatch_queue 'queued' do driver; VOLTAR->opt-in; AJUSTAR->envia deep link de preferências; VER NO MAPA->registra alert_feedback('clicked') + responde link de mapa; statuses[]->atualiza dispatch_queue e grava alert_feedback(delivered/read). driver_id é resolvido pelo telefone verificado, nunca confiado do payload.

POST/functions/v1/diesel-location-ingest

Recebe posição inferida. Body: {driver_id, lat, lng, heading_deg?, speed_kmh?, accuracy_m?, source:'live_share'|'last_route'|'checkin'}. Exige consent_location=true em diesel_alert_pref do driver, senão 403 e nada é gravado. Valida lat/lng (-90..90 / -180..180) -> 422 se inválido. Grava driver_location (expires_at curto) e dispara diesel-alert-evaluate de forma assíncrona (fire-and-forget; o 202 não espera a decisão). Retorna 202 {accepted:true, location_id}. Rate-limit por driver (default 1 ingest/30 s) -> 429.

POST/functions/v1/diesel-alert-evaluate

Núcleo de avaliação. Body: {driver_id, lat, lng, heading_deg?, speed_kmh?, source}. Resolve candidatos via ST_DWithin (raio do pref), filtra por preco_vigente (TTL+estado='confirmado'), direção de viagem (ST_Azimuth vs heading, exceção speed<10 km/h) e limiar (delta>=limiar_litro_centavos OU pct_abaixo>=limiar_pct). Ordena por economia desc, escolhe 1. Aplica dedupe (dedupe_key UNIQUE), teto_dia, frequencia_min, janela_silencio (America/Sao_Paulo), kill-switch e custo_teto. Se aprovado, INSERT diesel_alert(decision='fired') e dispatch_queue. Sempre grava diesel_alert (mesmo skip) para o funil. Retorna {decision:'fire'|'skip', skip_reason?, posto_id?, delta_litro_centavos?, economia_centavos?, alert_id}. Idempotente por dedupe_key.

POST/functions/v1/wpp-send-hsm

Worker chamado pela fila (pg_cron a cada 30 s) ou invocado pós-enqueue. Seleciona itens dispatch_queue com FOR UPDATE SKIP LOCKED WHERE status='queued' AND (next_attempt_at IS NULL OR next_attempt_at<=now()). Checa system_flag.kill_switch_global e custo_dia<custo_teto_dia_centavos ANTES de enviar; se bloqueado, mantém 'queued' sem incrementar custo. Chama Graph API POST /{PHONE_NUMBER_ID}/messages com template 'diesel_barato_rota'. Sucesso -> status='sending', wamid, custo_centavos; falha transitória -> attempts++, next_attempt_at = now()+backoff exponencial (1m,5m,25m), max 3; falha permanente -> 'failed'. Idempotente por alert_id (UNIQUE) — nunca reenvia.

PATCH/rest/v1/diesel_alert_pref?driver_id=eq.{id}

Atualiza preferências (opt_in, consent_location, raio_km, teto_dia, frequencia_min, janela_silencio_inicio/fim, limiar_pct, limiar_litro_centavos). Protegido por RLS (dono apenas). CHECK constraints rejeitam fora de faixa (raio 5..50 -> 400). Toda mudança de opt_in/consent_location grava consent_log (timestamp, finalidade, canal, base_legal, texto_versao) via trigger. opt_in=false -> honra opt-out imediato e cancela dispatch_queue 'queued' do driver.

GET/rest/v1/rpc/postos_proximos

RPC PostGIS para painel/debug. Params {lat,lng,raio_km,heading?,tipo?}. Retorna por posto: nome, preco_s10, preco_s500, media_rota, delta_litro_centavos, pct_abaixo, distancia_km, lado_pista, ttl_ok(bool), sentido_ok(bool). SECURITY DEFINER mas restrito a papel operação; não expõe driver_location. Usado para validar o gatilho manualmente.

GET/functions/v1/diesel-ops-dashboard

Painel admin (papel operação, exige JWT com role='operacao'). Retorna agregados: fila de disparos por status, taxa de entrega/leitura, CTR de VER NO MAPA, taxa de opt-out, custo HSM do dia vs teto, estado do kill-switch, % de cobertura de rota com preço vigente. APENAS agregados da mv_diesel_metrics; NUNCA lat/lng nem telefone identificável. PUT /functions/v1/diesel-ops-dashboard/kill-switch {on:bool} alterna system_flag (audita updated_by).

POST/functions/v1/diesel-lgpd-erase

Direito de exclusão (LGPD art. 18). Body: {driver_id}. Exige JWT do próprio driver ou papel operação com justificativa. Remove diesel_alert_pref, driver_location, alert_feedback do driver e desvincula o telefone; anonimiza diesel_alert/wpp_message_log (driver_id->null, payload com PII removida). Mantém consent_log mínimo anonimizado (hash do driver) como prova legal. Retorna {erased:true, tables:[...], at}.

6Frontend / fluxo

  • Tela 'Alertas de diesel' (PWA, dentro do app RODE): toggle de opt-in com texto de consentimento LGPD EXPLÍCITO (finalidade declarada: 'usar sua localização aproximada para avisar de posto barato na sua rota'; base legal: consentimento), slider de raio (5–50 km), seletor de teto diário (1–20), janela de silêncio (default 22:00–06:00). Estado via React com persistência otimista no PATCH /diesel_alert_pref e rollback em erro de validação. Exibe sempre 'Para parar a qualquer momento, responda PARAR no WhatsApp'. O toggle de localização é separado do de alertas (consent granular).
  • Componente de status do canal: mostra se o número WhatsApp está verificado/vinculado, data do último alerta recebido e economia acumulada no mês (soma de alert_feedback.valor_centavos). Reusa o card de economia do mock (src/data/diesel.ts: 700 L × R$0,27 = ~R$190). Quando sem dado fresco, exibe estado vazio explícito.
  • Mensagem proativa HSM 'diesel_barato_rota': corpo = 'Você está passando por {{cidade}} na {{rodovia}}. O {{posto}} está {{delta}}/L mais barato (R$ {{preco}}/L). No seu tanque dá pra economizar ~{{economia}}. {{distancia}} à frente.' Botões quick-reply: [VER NO MAPA] [AJUSTAR] [PARAR]. Exemplo renderizado com o mock: Ourinhos / BR-153 / Posto X / −R$0,27 / R$5,89 / ~R$190 / 6 km.
  • Fluxo conversacional de resposta: PARAR -> opt-out imediato + 'Alertas de diesel desligados. Responda VOLTAR pra religar.'; VOLTAR -> opt-in + confirmação; AJUSTAR -> deep link para a tela de preferências; VER NO MAPA -> registra alert_feedback('clicked') + responde link de mapa do posto (sem navegação própria).
  • Painel de operação admin (web, papel operação): fila de disparos do dia, status entrega/leitura por alerta, CTR de VER NO MAPA, taxa de opt-out, custo HSM acumulado vs teto com barra de progresso (alerta visual em 80%), botão kill-switch global com confirmação. Tabelas paginadas, somente agregados, rótulo 'atualizado até <timestamp>'. Sem exibir localização individual.
  • Tratamento de degradação: quando não há posição válida ou preço dentro do TTL, a tela mostra 'Sem dados frescos na sua rota agora' em vez de alerta velho; o gatilho não dispara (silencioso). Erros de rede no PATCH não perdem o estado local.

7Lógica de negócio

  • Regra de disparo (limiar): aprova se (media_centavos - preco_centavos) >= limiar_litro_centavos (default 15) OU pct_abaixo >= limiar_pct (default 3.0), onde pct_abaixo = (media_centavos - preco_centavos)/media_centavos*100. Caso RODE: média 616, posto 589 -> delta 27 (>=15) e 4,38% (>=3) -> dispara.
  • Economia por tanque (personalizada): economia_centavos = litros_tanque (perfil do veículo; mock 700) × delta_litro_centavos. Ex.: 700 × 27 = 18900 centavos -> formata 'R$ 190' (arredonda para a dezena de real mais próxima). Nunca usa valor genérico; sem litros_tanque no perfil, usa default conservador e marca economia como estimativa.
  • Filtro de direção: bearing = ST_Azimuth(geom_motorista, geom_posto) em graus; diff = menor ângulo entre heading_motorista e bearing (0..180). Aprova se diff <= 90 (posto à frente) OU speed_kmh < 10 (parado, ignora direção). Sem heading válido, exige distancia <= raio_km e usa o lado_pista compatível com a última rota; se nem isso, skip_reason='sentido_contrario'.
  • Filtro de candidatos: ST_DWithin(geom_motorista::geography, geom_posto::geography, raio_km*1000) AND posto.ativo AND existe preco_vigente (estado='confirmado' AND now()<coletado_at+ttl). Ordena por economia_centavos desc; escolhe o melhor 1. Sem candidato -> skip_reason='sem_candidato'.
  • Dedupe e teto: dedupe_key = md5(driver_id || posto_id || date_trunc('hour', now() AT TIME ZONE 'America/Sao_Paulo')). INSERT em diesel_alert com UNIQUE(dedupe_key) -> conflito = skip_reason='dedupe'. Recusa também se count(fired hoje) >= teto_dia ('teto_dia'), se now() < último_fired + frequencia_min ('frequencia'), ou se hora local dentro de [janela_silencio_inicio, janela_silencio_fim) ('janela_silencio', tratando intervalo que cruza meia-noite).
  • TTL do dado: sem preço vigente -> decision='skipped', skip_reason='preco_velho'. Nunca alerta com dado expirado — protege a confiança (risco crítico do PRD de produto).
  • Idempotência de envio: dispatch_queue.alert_id UNIQUE; worker consome com FOR UPDATE SKIP LOCKED; reentrega de webhook não duplica status (UNIQUE em wpp_message_log e alert_feedback). Retry com backoff exponencial (1m,5m,25m), max 3 tentativas, depois 'failed'.
  • Opt-out e cancelamento: PARAR/opt_in=false -> UPDATE diesel_alert_pref + consent_log(opt_out) + UPDATE dispatch_queue SET status='canceled' WHERE driver_id=$1 AND status='queued', tudo na mesma transação, em <5 s.
  • Kill-switch e teto de custo global: antes de wpp-send-hsm chamar a Graph API, lê system_flag.kill_switch_global e soma custo_centavos do dia; se kill_switch ON ou custo>=custo_teto_dia_centavos, deixa item 'queued' sem incrementar custo (skip_reason='kill_switch'/'custo_teto' no log de tentativa).
  • Feedback loop / calibração: alert_feedback (delivered/read/clicked/abastecido) alimenta CTR e economia realizada por motorista; se CTR persistentemente baixo, sugere subir limiar_pct/limiar_litro do motorista (ajuste, não automático sem revisão).

8Integrações

  • WhatsApp Cloud API (Meta Graph API v20+): envio do template HSM 'diesel_barato_rota', recebimento de status (sent/delivered/read/failed) e respostas de botão via webhook; verificação por VERIFY_TOKEN (GET) e assinatura HMAC-SHA256 (POST).
  • BSP alternativo (Twilio/Gupshup/Zenvia) atrás da interface WppGateway para portabilidade e fallback de número.
  • Provedor de map-matching/geocoding (Mapbox ou Google) para projetar a posição sobre a rodovia e refinar direção, acionado apenas quando o PostGIS local não basta.
  • Fonte de posição: compartilhamento de localização via mensagem do WhatsApp (location message), última cidade/rota conhecida do app RODE, ou check-in manual.
  • Módulo diesel-app (mesma plataforma RODE): consome preco_posto e a base de postos georreferenciada coletada por OCR/crowdsourcing — dependência de dados, não de runtime.
  • Perfil do veículo (módulo Calculadora/Cadastro): fornece litros_tanque para a economia personalizada.
  • pg_cron (Supabase): agendamento da purga de driver_location, agregação de media_rota, refresh da mv_diesel_metrics e tick do worker wpp-send-hsm.

9Segurança & LGPD

  • LGPD - geolocalização é dado pessoal: opt-in explícito com finalidade declarada e base legal (consentimento) registrado em consent_log com timestamp e versão do texto; consent_location separado de opt_in de alertas (granularidade). Sem consent_location=true, diesel-location-ingest retorna 403 e nada é persistido.
  • Minimização e retenção: driver_location.expires_at default now()+6h e job pg_cron de purga; guarda-se a posição só pelo tempo de avaliar o gatilho. NÃO se constrói histórico de trajeto persistente.
  • Opt-out imediato: PARAR honrado em <5 s, cancelando dispatch_queue 'queued' do driver e gravando consent_log(opt_out); nenhum alerta posterior até VOLTAR.
  • RLS por driver_id em diesel_alert_pref, driver_location, diesel_alert, consent_log e alert_feedback (políticas pgTAP-testadas). Motorista só lê/escreve o próprio dado; admin acessa apenas mv_diesel_metrics (agregados), sem lat/lng nem telefone identificável.
  • Verificação do webhook: GET valida VERIFY_TOKEN; POST valida X-Hub-Signature-256 (HMAC-SHA256 do corpo cru com APP_SECRET) ANTES de parsear — payload forjado é rejeitado com 401. driver_id resolvido pelo telefone verificado, nunca do payload.
  • Segredos fora do repo: WHATSAPP_TOKEN, PHONE_NUMBER_ID, APP_SECRET, VERIFY_TOKEN em variáveis de ambiente do Supabase; rotação periódica do token; .env no .gitignore.
  • Conformidade Meta: somente templates HSM aprovados para mensagem proativa fora da janela de 24 h; respeito ao tier/qualidade do número; opt-out reduz reclamações e protege o número de bloqueio.
  • Direito de exclusão (LGPD art. 18): diesel-lgpd-erase remove preferências, localizações, feedback e desvincula o telefone, anonimiza alertas e logs Meta, mantendo apenas consent_log mínimo anonimizado como prova legal. Toda escrita passa por Edge Function com checagem de dono; sem SQL livre exposto.

10Não-funcionais

  • Latência: < 60 s do recebimento da posição ao disparo do HSM (p95), medido por created_at(driver_location) -> created_at(dispatch_queue) com status>='sending'; webhook responde 200 em < 5 s (p99).
  • Custo controlado: teto diário de HSM por motorista (default 4) + teto de custo agregado global (system_flag) + kill-switch; custo por alerta em dispatch_queue.custo_centavos; alerta de operação quando custo_dia atinge 80% do teto.
  • Confiabilidade de entrega: outbox com retry exponencial (1m/5m/25m, max 3) e idempotência por alert_id; webhooks de status reconciliam o estado real (delivered/read/failed).
  • Frescor do dado: nunca disparar com preço fora do TTL (default 12 h); cobertura monitorada por % de rota com preço vigente no painel.
  • Disponibilidade: falha do map-matching degrada para PostGIS local sem derrubar o gatilho; falha da Cloud API mantém itens 'queued' sem perda.
  • Anti-spam: 1 alerta por posto por janela de hora (dedupe), janela de silêncio (America/Sao_Paulo) e frequência mínima entre alertas.
  • Escalabilidade geoespacial: índices GiST em posto.geom e driver_location.geom; postos_proximos p95 < 100 ms para base de 10k postos e raio 50 km.
  • Observabilidade: alert_id rastreável ponta-a-ponta; skip_reason tipado permite diagnosticar o funil; mv_diesel_metrics com refresh por pg_cron.

12Plano de teste

unit

  • limiar - caso RODE: media=616, posto=589 -> delta=27, pct=4,38% -> avalia('fire') (>=15c E >=3%)
  • limiar borda inferior: media=616, posto=605 -> delta=11, pct=1,79% -> 'skip' skip_reason='abaixo_limiar'
  • limiar borda OR: media=616, posto=601 -> delta=15 (==15) E pct=2,44% (<3) -> 'fire' (regra OR satisfeita pelo delta)
  • economia - tanque 700, delta 27 -> 18900c -> formata 'R$ 190' (arredonda dezena de real)
  • economia - tanque 400, delta 15 -> 6000c -> 'R$ 60' (personalização por tanque, não genérico)
  • direção à frente: heading=10°, bearing=20° -> diff=10 <=90 -> aprova; heading=10°, bearing=350° -> diff=20 <=90 -> aprova (normalização circular)
  • direção contrária: heading=10°, bearing=190° -> diff=180 >90 -> reprova ('sentido_contrario')
  • direção parado: heading=10°, bearing=190°, speed=5 km/h -> ignora direção -> não reprova por sentido
  • TTL expirado: coletado_at=now()-13h, ttl=720min -> não vigente -> 'preco_velho'
  • TTL no limite: coletado_at=now()-11h59m, ttl=720min -> vigente -> não bloqueia por TTL
  • dedupe_key: mesmo driver+posto+hora gera a mesma key (md5 estável)
  • janela de silêncio normal: hora local 23:30, janela 22:00-06:00 -> 'janela_silencio'
  • janela de silêncio cruzando meia-noite: hora local 05:30, janela 22:00-06:00 -> 'janela_silencio'; 06:30 -> fora da janela
  • frequência: último fired há 40 min, frequencia_min=60 -> 'frequencia'; há 70 min -> não bloqueia
  • assinatura webhook: HMAC-SHA256(corpo, APP_SECRET) calculado bate -> válido; corpo alterado em 1 byte -> inválido
  • formatação de centavos: 589 -> 'R$ 5,89'; 616 -> 'R$ 6,16'; 27 -> '−R$ 0,27'

integracao

  • ingest sem consent_location -> 403 e zero linhas em driver_location
  • ingest com consent_location e lat/lng inválidos (lat=120) -> 422, nada gravado
  • ingest válido -> grava driver_location e dispara evaluate; Posto X a 6 km à frente com preço vigente -> 1 linha em dispatch_queue('queued')
  • evaluate com teto_dia=4 já atingido hoje -> diesel_alert(decision='skipped', skip_reason='teto_dia'), sem dispatch_queue
  • evaluate sem candidato no raio -> skip_reason='sem_candidato'
  • wpp-send-hsm com kill_switch_global ON -> item permanece 'queued', Graph API não chamada (MSW), custo_dia não incrementa
  • wpp-send-hsm com custo_dia >= teto -> item 'queued', sem envio
  • wpp-send-hsm sucesso (MSW 200 com wamid) -> status='sending', wamid e custo_centavos preenchidos
  • wpp-send-hsm falha transitória (MSW 503) -> attempts=1, next_attempt_at futuro, status volta a 'queued'
  • webhook GET verify_token correto -> 200 com challenge; token errado -> 403
  • webhook POST assinatura inválida -> 401, nenhum efeito colateral
  • webhook status 'delivered' depois 'read' -> dispatch_queue atualizado e alert_feedback('delivered'),('read') sem duplicar
  • webhook reentrega do mesmo statuses[].id -> idempotente (UNIQUE), 1 só linha de log
  • PATCH diesel_alert_pref raio_km=4 -> 400 (CHECK 5..50); raio_km=8 -> 200 e updated_at muda
  • PATCH opt_in=false -> dispatch_queue 'queued' do driver vira 'canceled' e consent_log(opt_out) gravado
  • RLS: driver A tenta PATCH pref de driver B -> negado (0 linhas)

e2e

  • Jornada feliz: motorista dá opt-in + consent na tela -> compartilha localização (Ourinhos/BR-153 km312) -> recebe HSM 'diesel_barato_rota' com Posto X, R$5,89, −R$0,27/L, ~R$190, 6 km à frente, em <60 s (validado com Graph API mockada)
  • Opt-out: motorista responde PARAR -> recebe 'Alertas de diesel desligados...', opt_in=false em <5 s, próximo gatilho não dispara, consent_log registra opt_out; responde VOLTAR -> religa
  • VER NO MAPA: clique no botão -> alert_feedback('clicked') gravado, responde link de mapa, CTR sobe no painel admin
  • Dado velho: posição válida mas todos os preços do trecho fora do TTL -> nenhum alerta enviado (silencioso); painel mostra cobertura baixa naquela rota
  • Spam guard: dois postos baratos no mesmo trecho na mesma hora -> apenas 1 alerta (o de maior economia, Posto X), dedupe ativo no segundo
  • Janela de silêncio: posição válida às 23:30 local -> gatilho avaliado mas não dispara; mesma situação às 07:00 -> dispara

carga

  • postos_proximos com 10k postos e raio 50 km -> p95 < 100 ms com índice GiST
  • 1.000 ingests simultâneos -> fila absorve sem perder disparos e sem exceder teto de custo agregado (kill-switch/custo respeitados sob concorrência)
  • reconciliação de 5k webhooks de status -> sem duplicação (UNIQUE), log consistente
  • SKIP LOCKED: 5 workers concorrentes na dispatch_queue -> cada item processado por 1 único worker, zero envio duplicado

13Critérios de aceite

  • Dado um motorista com opt-in e consent_location ativos, tanque 700 L e posição em Ourinhos/BR-153, quando um posto a <=8 km à frente tem diesel >=R$0,15/L OU >=3% abaixo da média da rota com preço dentro do TTL, então ele recebe em <60 s (p95) o HSM 'diesel_barato_rota' com cidade, posto, preço (R$5,89), delta (−R$0,27/L), economia personalizada (~R$190) e distância (6 km).
  • Dado um motorista sem consent_location ativo, quando uma posição é enviada a diesel-location-ingest, então a resposta é 403, nenhuma linha é persistida em driver_location e nenhum alerta é gerado.
  • Dado um motorista que responde PARAR no WhatsApp, quando a mensagem é recebida, então opt_in vira false em <5 s, todos os itens 'queued' do driver em dispatch_queue viram 'canceled', consent_log registra opt_out e nenhum alerta é enviado até ele responder VOLTAR.
  • Dado que todos os preços do trecho estão fora do TTL (default 12 h), quando o gatilho é avaliado, então diesel_alert é gravado com decision='skipped' e skip_reason='preco_velho' e nenhuma mensagem é enviada.
  • Dado que o motorista já recebeu o teto diário de alertas (default 4), quando um novo posto barato é detectado, então o disparo é recusado com skip_reason='teto_dia' e custo_dia de HSM não aumenta.
  • Dado um posto barato no sentido contrário ao heading do motorista (diferença angular >90°), quando o gatilho é avaliado, então não dispara (skip_reason='sentido_contrario'), exceto se speed_kmh<10 (motorista parado), caso em que a direção é ignorada.
  • Dado o mesmo posto, mesmo motorista e mesma janela de hora (America/Sao_Paulo), quando dois gatilhos ocorrem, então apenas um alerta é enviado porque o segundo INSERT viola dedupe_key UNIQUE (skip_reason='dedupe').
  • Dado o webhook da Meta, quando chega POST com X-Hub-Signature-256 inválida, então a resposta é 401 e nenhum efeito colateral ocorre; e quando o mesmo statuses[].id/messages[].id chega duas vezes, então o processamento é idempotente (1 só linha de log e de feedback).
  • Dado o kill_switch_global ativado (ou custo_dia >= teto), quando há itens 'queued' na fila, então a Graph API não é chamada, os itens permanecem 'queued' e custo_dia não incrementa.
  • Dado um alerta entregue e clicado em VER NO MAPA, quando os webhooks/clique chegam, então alert_feedback registra delivered/read/clicked sem duplicar e o painel de operação reflete CTR e economia estimada (litros × delta).
  • Dado o PATCH /diesel_alert_pref com raio_km fora de [5,50], quando enviado, então a API responde 400 (violação de CHECK) e nenhuma preferência é alterada.
  • Dado postos_proximos com base de 10k postos e raio 50 km, quando consultado, então o p95 de resposta é < 100 ms (índice GiST).
